## A. The modern Android port: emileb/OpenJK @ `master_mobile` (the key artifact)

The current shipping app is **Psi Touch** (`emileb/Psi_Touch`, pushed 2026-07), which bundles
Jedi Outcast SP, Jedi Academy SP+MP, and Dark Forces. Its engine is a gitlink pinned to the tip of
**`emileb/OpenJK` branch `master_mobile`** — a fork of the same JACoders/OpenJK we build.

**Delta vs upstream: 68 files, +4,283/−117, merge-base `8cce3ea2` (2026-05-13) — only 2 commits
behind current upstream master.** Discipline is excellent: renderer strictly `#ifdef USE_GLES1`,
platform strictly `#ifdef __ANDROID__`/`if(ANDROID)`, SP-vs-MP via `OPENJK_MP`/`JK2_MODE`.
Breakdown: `code/` +1,495 (28 files), `codemp/` +1,378 (mirror), `mobile/` +1,160 (new touch
bridge), `shared/` +115, CMake ~+150, `codeJK2/` +4/−3 (JK2 game logic essentially untouched).

### Renderer (`USE_GLES1`)
In-tree OpenGL ES 1.1 fixed-function port of rd-vanilla. No gl4es, no shaders, no GLES2 path;
rend2 explicitly not ported. Links `GLESv1_CM` + `EGL`. Per-feature:
- `qgl.h`: `<GLES/gl.h>` (gated on `__ANDROID__`), `GLdouble→GLfloat`, `glOrtho→glOrthof`,
  `glClearDepth→glClearDepthf`, `glClipPlane→glClipPlanef`, `qglColor3f→glColor4f(...,1)`,
  inline `glColor4ubv`, ARB multitexture → core GLES entry points, missing tokens defined manually.
- Immediate mode: every `glBegin/glEnd` site (~24 in SP renderer) rewritten as client vertex
  arrays + `glDrawArrays`/`glDrawElements`, with client-state save/restore. ~16 files, mechanical.
- **Stencil shadows kept working**: tr_shadows.cpp (+286) converts volumes to indexed arrays,
  plus `_STENCIL_SHADOW_OPT` caching silhouette indices per frame. JK2 shadow fix in `bf587afb`.
- **Dynamic glow compiled out** (needs ARB programs; `r_DynamicGlow` defaults 0 — same as our
  macOS reference default).
- Cinematics (`RE_StretchRaw`): vertex-array quad, internal format forced `GL_RGBA`,
  `GL_CLAMP→GL_CLAMP_TO_EDGE`.
- `glReadPixels`: RGBA temp buffer repacked to RGB (screenshots work); stencil/depth readback out.
- Textures: sized/compressed internal formats removed; `Upload32` always `GL_RGBA`.
- Gamma: no hardware gamma on GLES → new `RB_Brightness()` fullscreen blend pass per swap.
- Alpha test + texture-env (GL_ADD/GL_COMBINE): **zero changes — core in GLES 1.1.**

### Module linking
**Still fully dynamic.** On Android the engine itself is a SHARED lib entered from JNI
(`main_android()`); game (`libjospgame.so`) and renderer (`librdjosp-vanilla.so`) are dlopen'd
from the APK native-lib dir (legal on Android). "No PK3 unpacking or fs_* search path on Android."
**The branch never confronts static linking — no help for the iOS no-dlopen constraint.**

### Platform layer
External Beloko wrapper repos supply SDL2 fork + JNI glue + touch controls:
`AndroidCore_OpenTouch` (Java host), `SDL2_OpenTouch` (vendored SDL 2.0.12, ~14 files patched
behind `OPENTOUCH_SDL_EXTRA`: surface-resume fixes, swap-callback hook so touch overlay draws
before swap), `Clibs_OpenTouch`, `MobileTouchControls` (GL overlay), `SAFFAL` (scoped-storage
shim — irrelevant on iOS), `AudioLibs_OpenTouch` (openal-soft + oboe). Native blocking main loop
on its own thread. Touch input injected via SDL internal `SDL_SendKeyboardKey` so buttons stay
remappable; analog move drained on engine thread by new `CL_AndroidMove()` from `CL_CreateCmd`;
SP menus get `UI_MobileSetCursor()` tap-to-select. EAX carved behind `USE_OPENAL_EAX`.
JK2 SP is first-class (`BuildJK2SP* ON` on Android; force-power wheel handles JK2_MODE).

### Apple specifics
**None.** Zero `__APPLE__`/iOS lines in the diff. The qgl.h GLES include is gated `__ANDROID__`.

## B. Lineage: the 2013–2014 hand-GLES ports (historical, superseded)

- **xLAva/JediAcademyLinux + ptitSeb's Pandora commit (2013)** — origin of the GLES1 approach.
  One commit, 73 files, +3,565/−1,147; `HAVE_GLES` ifdefs; same per-feature story as above
  (immediate-mode→arrays, format conversions, glow off). ptitSeb later wrote gl4es partly to
  avoid ever hand-porting again. ARM fix categories: alignment (RoQ blits→memcpy),
  double→float sweep, x86 asm removal, optional NEON math. JKA only.
- **emileb/JK3 (2014)** — Android wrap of that Pandora port (pre-OpenJK Raven source, JKA).
  Renderer statically in the engine (Raven SP always was; OpenJK modularized it later); game+cgame
  in one dlopen'd .so with `-fvisibility=hidden` + 3 exported symbols (`GetGameAPI`, `dllEntry`,
  `vmMain`). Java-driven frame callbacks (`PortableFrame` per frame) — superseded by
  master_mobile's native loop. Notable: `-fsigned-char` required (id Tech 3 assumes signed char;
  ARM defaults unsigned — same applies to iOS/arm64 — though clang arm64 Darwin defaults signed,
  verify). Avoid: their `-Ofast -ffast-math` flags (savegame/physics determinism risk).
- **emileb/Open_IOS (2016)** — abandoned 2-month experiment staging SDL2 2.0.3 + DarkPlaces +
  TouchControls for iOS. Proves TouchControls once compiled for iOS; otherwise stale.

## C. iOS prior art (no JK2/JKA iOS port exists — we'd be first)

- **ioquake3-family on iOS** (tomkidd/Quake3-iOS, RTCW-iOS): run QVMs interpreted — no native
  game-module precedent. Renderer: opengl1 + Zack Middleton GLES1 patches on **SDL2's iOS
  backend** (EAGL). tomkidd/DOOM3-iOS (dhewm3) is the nearest native-C++-game-in-app precedent.
- **Static-link precedent**: Quake3e/ioq3 `USE_RENDERER_DLOPEN=0` — engine calls `GetRefAPI`
  directly, no symbol tricks needed for the renderer half. **jk2mv** (JK2 MP) compiles the
  renderer into the client and calls `GetRefAPI(REF_API_VERSION, &ri)` directly — proves the
  OpenJK-era refapi tolerates static linking.
- **dlopen nuance**: iOS forbids loading unsigned code from the sandbox, but dylibs **embedded
  in the app bundle and signed with the app identity are dlopen-able** (standard practice).
  So bundle-embedded `jospgame`/`rdjosp-vanilla` dylibs + patched search paths is a viable
  low-effort fallback to full static linking, including under AltStore/SideStore re-signing.
- **Desktop/mobile GL on iOS in 2026**:
  - Apple's **OpenGLES.framework: deprecated since iOS 12 but still present and functional
    through iOS 18/26**, including **ES 1.1 fixed-function contexts** via EAGL. SDL2's iOS
    video backend creates ES 1.1/2.0/3.0 contexts natively.
  - **ANGLE** (GLES-on-Metal, Apple co-developed, ships in WebKit): has a **GLES 1.1 frontend**
    implemented on ES 3.0 → fixed-function code can run on Metal. MetalANGLE standalone fork is
    unmaintained (merged upstream). Needs hand-rolled EGL glue next to SDL on iOS.
  - **gl4es on iOS is shipped practice**: PojavLauncher/Amethyst run desktop-GL Minecraft through
    gl4es→ANGLE→Metal on sideloaded iOS. And **PortMaster ships Jedi Outcast itself** on GLES-only
    ARM handhelds as stock `openjo_sp` + unmodified rd-vanilla through gl4es — proof for this
    exact game.
  - MoltenGL: dormant, GLES2-only, skip.
- **OpenJK ecosystem**: upstream has no GLES renderer (issue #1091). glKarin/diii4a contains an
  independent second GLES1.1 conversion of rd-vanilla (`_GLES` ifdefs) — useful cross-reference
  when debugging. JKSunny/EternalJK Vulkan renderer is JKA-MP-only (future MoltenVK curiosity).
  Team Beef's **JK XR** (Quest VR, OpenJK-based, sideloaded) is adjacent Android prior art.

## D. What this means for our port (synthesis)

1. **Renderer**: the fear that rd-vanilla needs a graphics-API *rewrite* is dead. Three
   ship-proven paths, cheapest first:
   (a) cherry-pick `master_mobile`'s `USE_GLES1` patch (≈1,000 lines for SP, GPLv2, near-tip
   merge-base) → SDL2 ES1.1 context on OpenGLES.framework — no translation layer, no EGL glue;
   risk = Apple someday removing a framework that's survived 8 years of deprecation;
   (b) same patch on ANGLE GLES1-frontend→Metal — future-proof, needs custom EGL glue;
   (c) unmodified rd-vanilla through gl4es→ANGLE→Metal (PortMaster/Pojav precedent) — zero
   renderer edits, two translation layers to debug.
   (a) and (b) share the same in-tree patch; (a) upgrades to (b) without rework.
2. **Linking**: nobody has solved it for us, but the seam is tiny (`GetGameAPI` + `GetRefAPI`
   function tables; 3 exported symbols per module; `-fvisibility=hidden` already on). Plan:
   static-link via per-module `ld -r` prelink + localize-all-but-entry-symbol + direct calls
   behind a small define (jk2mv precedent), built and verified on macOS first. Fallback that
   needs almost no work: embedded signed dylibs in the app bundle, dlopen'd by absolute path.
3. **Input/lifecycle** (later phases): master_mobile's `mobile/game_interface.cpp` pattern
   (SDL key injection + `CL_AndroidMove` + `UI_MobileSetCursor` tap-to-select + swap-callback
   overlay hook) is the design to imitate; their TouchControls libs are Android-tied.
4. **Audio**: SP's OpenAL path works on mobile with openal-soft (`USE_OPENAL_EAX` carve-out);
   SDL audio is the fallback. iOS has system OpenAL (deprecated) or openal-soft built statically.
5. **Assets**: their folder-drop UX maps 1:1 onto iOS Documents + `UIFileSharingEnabled`;
   none of their SAF machinery is needed.
